## What to research

### 1. Skill Eval Harness & Test Suite Architecture
- **Lightweight Test Cases:** JSON/YAML schema defining test prompts (5 happy path + 5 negative path minimum per skill), execution language/workspace, expected trigger behavior (`should_trigger: bool`), and assertions.
- **Cheap Assertions over Heavy LLM Judges:** Prefer deterministic regex checks on trajectories/outputs (correct SDK methods, proper flags, forbidden legacy patterns) before falling back to LLM-as-a-judge rubrics.
- **Isolated Execution Environments:** Run skill evals in pristine, isolated workspaces to prevent agents from cheating or accessing stale chat histories.
- **Multi-Trial & Multi-Harness Testing:** Run 3–6 trials per test prompt across different agent harnesses (Anti-gravity, Cursor, Claude Code, custom wrappers) to distinguish model non-determinism from skill flaws.

### 2. Progressive Disclosure & Context Budgeting
- **Strict Word Limit:** Keep primary `SKILL.md` files below 500 words. Layer detailed implementation details into secondary reference files (e.g. AWS vs GCP guides) loaded conditionally.
- **System Prompt Overhead Awareness:** Skill descriptions are loaded into the system context on every turn (~100–200 tokens). Vague or bloated descriptions incur permanent cost and risk triggering failures (~50% of skill failures stem from bad descriptions).
- **Directive Language:** Use explicit instructions ("Use the Interactions API when building chat applications") rather than passive recommendations ("Interactions API is recommended...").

### 3. Skill Taxonomy & Lifecycle Management
- **Capability vs. Preference Skills:**
  - *Capability Skills:* Temporary scaffolding to bridge model knowledge gaps. Must be monitored for retirement as base models improve.
  - *Preference Skills:* Durable standards for company-specific, domain-specific, or architectural patterns. Require eval protection against model regressions.
- **No-Op Elimination:** Detect and strip non-functional instructions ("Write clean, maintainable code", "Make implementation easy to read") commonly generated by AI prompt tools.
- **Continuous Ablation Testing:** Periodically evaluate agent task completion **with and without** the skill enabled. If the base model matches skill performance without the skill loaded, retire the skill to save tokens and latency.
- **Retained Evals as Safeguards:** Retain the eval test suite after retiring a skill to detect future model regressions or behavior drift.

## Why it matters for Gaia Research & Skill Tree

1. **Direct Alignment with GSB (Gaia Skill Bench):** Enhances our benchmark ingest engine (`scripts/validate-submissions.ts`, `gsb-submission.schema.json`) with standardized eval harness structures and ablation scoring methodologies.
2. **Internal Skill Audit & Hardening:** Provides an actionable blueprint for auditing and optimizing Gaia's internal skills (`.agents/skills/`, `.pi/skills/`) to eliminate no-ops, enforce <500-word thresholds, and write negative test cases.
3. **Ecosystem Thought Leadership:** Establishes Gaia as the premier advocate and verification platform for skill quality, progressive disclosure standards, and skill eval harnesses in the AI agent ecosystem.
