**Epic:** #89 · **Issue:** #86 · **Depends on:** #85 (landed) + Plan A (Yggdrasil II sync hardening — see note on ordering) · **Target branch:** PR into `epic-89/full-registry-discoverability` (integration branch head)

## 0. The one-paragraph problem (from #86, re-verified)

100% of *deterministic* skill discovery in Infinite Skill Craft is hand-authored today: `lib/craft/starter-recipes.ts` (a 49-entry manual tech tree from the 4 seeds `/prompt /code /web /data`), while `data/craft/recipes.json` holds registry-derived fusions (57 usable 2-input) of which **a BFS from the 4 seeds reaches zero** — their prerequisite pairs need intermediate skills a player can never craft from the seeds. Net: every new registry skill needs a human to hand-write a bridge to be reachable. The fix is to make reachability a **derived, build-time computation over the real registry graph**, so new skills get a path automatically on the next sync, at zero request-time cost.

## 1. What Yggdrasil II changes for THIS task (why the graph is now richer)

The pre-Yggdrasil analysis in #86 measured reachability using `recipes.json` (parsed from `combinations.md`) ∪ `starter-recipes.ts`. Yggdrasil II gives us a **better edge source**: every `registry/nodes/fusion/*.json` carries **inline `prerequisites: string[]`** (skill IDs). Verified on the staging branch:

- 130 fusion nodes total. Prereq-count distribution: **57 with exactly 2**, 55 with 3, plus a tail (4,5,6,…,31 — one node has 31).
- These inline prereqs are a **superset** of what `combinations.md` encodes (spot-checked: `advanced-swarm-coordination` has inline prereqs `swarm-topology-management + multi-agent-debate` but **no `combinations.md` row**).

**Prerequisite:** Plan A (#ygg2-sync-hardening) is what surfaces those inline `prerequisites` into `data/craft/skills.json`. This plan (B) consumes them. **Coordinate on ordering:** if Plan A has not merged into the integration branch when you start, either (a) rebase on it once it lands, or (b) read the inline prereqs directly from the sync's in-memory node list by having your reachability step run inside/after the sync. **Do not duplicate the registry read** — consume the sync's output (`skills.json` + `recipes.json`), which is the whole point of #86 ("derived at build time, next sync picks up new skills automatically"). Flag the dependency explicitly in your PR and, if you had to stub the fusion-prereq field because Plan A wasn't merged yet, say so loudly.

## 2. Design: a build-time reachability derivation

### 2.1 The graph
Nodes = skill IDs. A fusion node `F` with `prerequisites: [p1..pn]` means the directed hyperedge "**all** of p1..pn craftable ⟹ F craftable". Seeds = the 4 primitives (`/prompt /code /web /data` → their slugs; confirm the exact seed slugs the UI worker sets — grep the fuse route/worker, do not assume). Basic nodes with empty prereqs that are *also* reachable only matter if they're seed-adjacent; treat the 4 seeds as the sole roots unless the code says otherwise.

### 2.2 The closure (AND-reachability over a hypergraph)
Standard fixpoint: start with `reachable = {seeds}`; repeatedly add any node all of whose prerequisites are in `reachable`; iterate to fixpoint. This correctly handles the 2-, 3-, …, 31-prereq fusions uniformly (the old `findRecipe` only understood 2-input pairs and ignored the ~72 multi-prereq entries — #86 explicitly says investigate those; the hypergraph closure subsumes them). Include `combinations.md`-derived recipes as additional edges where they add prereq sets the inline data lacks (union the two edge sources; dedup by result id).

### 2.3 Output artifact: `data/craft/bridges.json` (new, generated)
Generated by the sync (extend #85's script) or an adjacent build step that reads the sync output. Contents (shape is yours to finalize, but must include):
- The **derived reachable set** (skill ids reachable from seeds) and, per reachable fusion, the **prerequisite set** that unlocks it (so the client can present a real derived path, replacing hand-authored bridges as the *mechanism*).
- A **reachability report**: `{ totalRegistrySkills, reachableCount, reachablePct, unreachableCount, generatedAt, seedSlugs }` plus the list (or count + sample) of unreachable skills. This is the DoD's "regenerated every sync" report.
- Keep it anti-bloat consistent with the other `data/craft/*` (ids/slugs, not full records/URLs).

### 2.4 `starter-recipes.ts` — demote from mechanism to overlay
Per #86: the derived structure becomes the **source of truth for reachability**; the hand-authored tier-1/2/3 names stay only as **flavor/copy overlay** (recognisable result names, curated descriptions) layered on top — or are retired. Do **not** delete it blindly; keep whatever the tests + UI rely on for recognisable naming, but ensure reachability is no longer *sourced* from it. Wire `findRecipe`/the fuse route to consult the derived set. Preserve existing guarantees (tier-1 seed combos still resolve; no duplicate pairKeys).

### 2.5 The genuine-gap decision — HUMAN GATE, do not auto-resolve
Where the real graph doesn't span from seeds to a skill, #86 says decide **explicitly**: (a) synthesize a minimal *derived* bridge (auto-generated shortest missing path — never hand-picked flavor) or (b) accept the skill is emergent/LLM-only (#87) and **document the boundary**. This is a **founder product call**, not an engineering default. Your job: compute and *present* the honest number and the unreachable list with both options costed; **flag it in the PR and get an explicit yes/no before merge.** A script that reports "100% reachable" by silently synthesizing nonsense bridges is worse than an honest partial number — the plan says so explicitly.

## 4. Tests (must add / update)

- **Fixture-driven reachability test:** a tiny fake registry (seeds + a couple basic + a fusion whose prereqs are seed-reachable + one deliberately unreachable fusion) → assert the closure includes the reachable fusion, excludes the unreachable one, and the report counts are exact. Then **add one new skill to the fixture** and assert it becomes reachable with zero code change (this is the DoD's headline proof).
- **No-regression:** `starter-recipes.test.ts` and the fuse route test stay green; tier-1 seed combos still resolve; `pairKey` uniqueness preserved.
- Determinism: closure output is stable across runs (sort ids; no `Date.now()`-dependent ordering in the reachable set — `generatedAt` is fine as a field, just not as a sort key).
- `npx tsc --noEmit` clean; `npx vitest run` green (baseline 166/166 after #85 — count will grow).
