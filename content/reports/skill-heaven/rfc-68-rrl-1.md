# Designing a Tiered Agent-Computer Interface for Tool Exposure

## Executive synthesis

The literature does **not** support a simple “more tools is better” story. What it supports is a separation between a **large global registry** of reachable tools and a **small active frontier** of tools exposed to the model at any instant. Across recent work, architectures that scale well do so by **retrieval, hierarchy, graph routing, or parameterized tool memories**. They do **not** win by dumping hundreds or thousands of tool schemas directly into the prompt. AnyTool explicitly reports that putting all 16,000+ APIs into GPT-4 exceeds even a 128k context window, while ToolNet argues that naive in-context listing becomes impractical both because of token cost and because selection quality degrades as the tool set grows. ToolRet then shows that even strong retrievers still struggle on 43k-tool retrieval, and that retrieval quality materially affects downstream pass rate.

There is also now strong evidence that **irrelevant context harms reasoning before you hit the hard context limit**. SWE-agent shows that overly human-oriented, granular interfaces and unbounded history degrade software-engineering performance; long-context RAG work finds an “inverted-U” where more retrieved passages initially help and then hurt; and “Context Length Alone Hurts LLM Performance Despite Perfect Retrieval” shows that sheer input length can degrade performance by 13.9% to 85% even when retrieval is perfect and even when irrelevant material is masked or replaced with whitespace. In other words, your “Skill Heaven” intuition is not merely aesthetic; it is well aligned with the current evidence base.

My synthesis for ACI design is therefore this: **never let “Max” mean “serialize the whole tool universe into the context window.”** Let “Max” mean **maximal reachability** through a routed system. The research points toward two practical limits: keep the **local branching factor** small, often around **3 to 6 tools per decision**, and keep the **active prompt-exposed candidate pool** in the **low tens to low-sixties**, with 64 looking like a realistic saturation point in large-API settings. That recommendation is an engineering inference from AnyTool’s ablations, ToolNet’s successor sparsity, and long-context RAG studies, rather than a formal field-wide law.

## What the literature says about maximal and minimal tool exposure

Recent tool-use systems overwhelmingly converge on **dynamic narrowing** rather than raw maximal exposure. ToolLLM introduced ToolBench and ToolLLaMA around a corpus of **16,464 real-world APIs**, but even there the practical pipeline includes a **neural API retriever** that recommends relevant APIs before the model solves the task. ToolLLM’s broader contribution is therefore not “all tools in context,” but a retrieval-plus-reasoning pipeline over a massive API pool.

AnyTool sharpens that point. It starts from the same general regime of **16,000+ APIs** and states plainly that integrating all of them into GPT-4 at once exceeds the model’s maximum context length, even at **128,000 tokens**. Its answer is a **hierarchical retriever** plus a solver plus a self-reflection loop. On its more realistic AnyToolBench setting, a “plain” GPT-4 retriever that sequentially scans the API space in **33 groups of 500** APIs reaches only **14.0%** pass rate, while AnyTool reaches **73.8%**. The ablations are even more informative for your slider design: removing the hierarchical structure drops pass rate from **58.9/63.2** to **22.4/15.8** on G2-I/G3-I; removing self-reflection drops it to **19.6/15.8**; a candidate pool of **64** APIs “nearly reaches saturation”; and a lower-level tool agent does best when managing **5** tools, while **10** tools is worse.

ToolNet reaches a similar conclusion from a different angle. Instead of retrieval, it uses a **tool graph** that routes the model only to plausible successors. Its key empirical observation is that in ToolBench, **around 80% of tools have fewer than 6 successor tools called**. That is a very practical signal: the real action space of multi-step tool use is often **sparse and local**, not globally dense. ToolNet therefore exposes only successor tools rather than the full tool universe. On ToolBench with **3,992 APIs**, it reports a **0.75** win rate, beating ReAct and ToT variants, while using fewer tokens than Reflexion baselines.

ToolRet then shows why this matters. It builds a **43k-tool**, **7.6k-task** benchmark and finds that conventional IR models “aren’t tool-savvy”: even strong retrievers struggle, and replacing the officially annotated toolset with retrieved tools causes the agent’s end-to-end performance to **substantially drop**. The paper gives a concrete example in ToolBench-G1 where GPT-3.5 falls to **50.60** pass rate using tools retrieved by one retriever, a drop of **11.40** points versus oracle tools. So the academic consensus is not “retrieval solves scale”; it is “**retrieval is mandatory, and retrieval itself is now a first-class bottleneck**.”

There is one important exception to the retrieval-first pattern: **ToolGen**. ToolGen reports results on **47,000+ tools** without an external retrieval step by representing each tool as a **unique virtual token** and integrating tool knowledge into model parameters. That is a genuine counterexample to the idea that tool scaling must always be handled by runtime retrieval. But it is **not** a success case for raw maximal prompt exposure. It escapes the context-bloat problem by moving the tool universe from **prompt space** into **parameter space**. For your ACI, that matters because it suggests a future path—tool embeddings, learned tool identifiers, compiler-like tool vocabularies—but it does **not** validate a “Hell Mode = all schemas in context” architecture.

The clearest answer to your “when does tool bloat start to hurt?” question is therefore a layered one. At the **global registry** level, systems can now reach thousands or tens of thousands of tools if they use retrieval, graph routing, or parameterized tool memories. At the **active exposure** level, the pain starts much earlier. AnyTool’s inner tool agents work best around **5** tools and start degrading by **10**; ToolNet shows real next-step choice sets are usually **under 6**; AnyTool’s prompt-level candidate pool saturates around **64**; long-context RAG performance often turns downward as the number of retrieved passages rises from **tens** toward **hundreds**; and MCP-specific stress tests show selection accuracy collapses when naive all-tool prompting faces very large registries. The practical implication is that your design variable should not be “total installed tools,” but **simultaneously serialized tools** and **simultaneously plausible choices**.

## What ACI design does to reasoning

SWE-agent is one of the strongest papers for your problem because it cleanly isolates **interface design** as a performance variable. The core claim is that language models are a new class of end user and need their own interface, not a direct inheritance of human UIs like the raw Linux shell. SWE-agent’s ACI replaces a granular shell action space with a **small set of simple actions** for viewing, searching, and editing files; adds **specific, concise feedback**; and introduces **guardrails** such as edit validation. On SWE-bench Lite, the full SWE-agent configuration resolves **18.0%** of tasks, versus **11.0%** for the shell-only agent. The paper’s design principles are almost tailor-made for your “Skill Heaven” idea: actions should be simple, compact, informative, and guarded.

The ablations are especially revealing. A **summarized** search interface achieves **18.0%**, while an **iterative** search interface—closer to a human IDE pattern with next/prev result navigation—drops to **12.0%** because agents inspect search results exhaustively until they burn through cost budget or context window. A **100-line** file viewer works best at **18.0%**, while a **full-file** view falls to **12.7%**. Keeping only the **last 5 observations** works better than carrying the **full history** (**18.0%** vs **15.0%**). Edit guardrails also matter: the editing configuration without linting drops from **18.0%** to **15.0%**, and “No edit” falls to **10.3%**. This is direct evidence that both the **volume** and the **shape** of the command/feedback surface affect System 2 performance.

CodeAct reinforces the same lesson from a different direction. Instead of offering many pre-defined tool schemas or JSON action formats, it collapses action taking into **executable Python code**, i.e., a unified action space. Across 17 LLMs on API-Bank and a new benchmark, CodeAct reports **up to 20% higher success rate** than standard tool/action formats. That is another way of saying that reducing the model’s need to reason over a fragmented action ontology can improve performance. In your ACI terminology, this is a powerful argument for a compact **Native** tier that favors a small number of composable primitives over a sprawling menu of narrow skills.

There is also increasingly direct evidence that **stripping away irrelevant tools or tool-like context improves reasoning**, though the effect depends on model family and task. GSM-DC shows that irrelevant context significantly disrupts both reasoning path selection and arithmetic accuracy. Long-context RAG studies show that performance often follows an **inverted-U** as retrieved context expands, with extra passages eventually reducing precision and distracting the model. And the EMNLP 2025 paper on context length finds that simply making the input longer can hurt math, QA, and coding performance even with perfect retrieval. These findings do not say “fewer tools is always better.” They say something more useful: **when the task is primarily architectural or reasoning-heavy, every non-essential schema consumes cognitive bandwidth**.

One subtle but important caveat comes from ToolBeHonest and more recent tool-hallucination work. ToolBH finds that top models still perform poorly overall on hallucination-sensitive tool tasks, and that the **primary error source is solvability assessment**, not merely syntax. It also reports that **verbose replies** can hurt open-weight models, while some proprietary models benefit from longer reasoning. More recent work, including *The Reasoning Trap*, argues that stronger reasoning can actually **increase tool hallucination** if the model is rewarded for “think-then-act” behavior without simultaneous reliability alignment. So yes, a move toward Heaven can improve reasoning—but only if Heaven means **better-calibrated action choice**, not just “less thinking.”

## A practical slider for discrete tool tiers

The literature suggests that your slider should be defined along **four controllable quantities**, not one. Let **R** be the total registry size, **A** the number of tool schemas actually exposed in the prompt for the current step, **B** the local branching factor the model must choose from, and **H** the amount of tool-use history carried forward. The research consistently says that **R can be very large**, but **A, B, and H must stay bounded**. ToolNet’s real trajectories are sparse; AnyTool saturates around 64 active APIs and prefers 5-tool micro-agents; SWE-agent performs worse with full history and full-file views; long-context work shows that raw length itself is harmful.

A good five-rung design for an ACI tiering system would look like this:

- **Off**: **A = 0, B = 0**. No external tools. Use only the base model and whatever fixed system instruction you already trust for architecture, triage, code review, or design discussion. This is the right setting when the task is concept formation, decomposition, or explanation rather than state-changing execution. The case for this tier comes from SWE-agent’s full-history ablation, long-context degradation, and tool-overuse work showing that unnecessary tools waste budget and can reduce accuracy.

- **Low**: **A up to about 4, B up to about 3**, read-only or dry-run primitives only. Think “repo search, file read, diff view, test discovery,” but no mutating external APIs. This is a “guided Heaven” mode: enough tooling to ground the reasoning, not enough to let the model thrash. The literature does not give an exact low-tier number, but ToolNet’s successor sparsity and AnyTool’s micro-agent result both support keeping the local choice set very small.

- **Native**: **A up to about 16, B up to about 5**. This is the SWE-agent / CodeAct / coding-harness sweet spot: a compact set of composable, high-leverage primitives for search, edit, execute, inspect, and patch, with concise feedback and guardrails. If your product is an ACI for coding agents, this should probably be the default tier. It aligns with SWE-agent’s result that a small, LM-centric action set beats the raw shell, and with CodeAct’s result that a unified action space can outperform more fragmented tool interfaces.

- **High**: **A up to about 32, B still capped around 5**, with mandatory **family-level retrieval or router selection** before exposing tools. At this tier you can add external APIs, MCP servers, or domain plugins, but only after an intent classifier or tool-family retriever has narrowed the space. AnyTool’s results strongly support hierarchy here; ToolRet shows that retrieval becomes a first-order systems problem; and RAG-MCP shows that retrieval-based MCP selection outperforms all-tools prompting while cutting prompt tokens.

- **Max**: **R may be unbounded, but A should still be capped around 64 and B around 6**. “All tools on” should mean “all tools are addressable by the router,” **not** “all tools are serialized into the model context.” This tier should only be reachable through a hierarchical router, tool graph, or equivalent staged retrieval process, plus reflection and verification. The number **64** is not magic, but it is one of the clearest empirical saturation points in AnyTool, and it sits in the same general regime suggested by ToolNet’s sparse branching and long-context RAG’s inverted-U behavior.

The semantic difference between rungs should therefore be **not merely more tools**, but a change in **where complexity lives**. Lower tiers should concentrate complexity in the model’s reasoning and keep the interface nearly frictionless. Higher tiers should **move complexity out of the prompt and into the routing substrate**: tool families, hierarchical search, graph edges, capability tags, learned retrieval, and policy checks. Past a small curated set, I would treat **flat prompt injection as a research anti-pattern**, not as a scaling strategy. The strongest evidence for that claim is that AnyTool’s flat/plain retriever collapses, ToolRet shows retrieval quality is bottlenecking end-to-end pass rate, and ToolGen’s main contribution is precisely to avoid prompt serialization by moving tools into model parameters.

## Firebreaks for hell mode

If a user explicitly asks for “Hell Mode,” the first firebreak should be a **tool-necessity gate before tool exposure**, not after the first bad call. ToolBeHonest shows that a major share of failures comes from **misjudging solvability**, and SMART identifies a persistent **tool overuse** bias where models invoke tools more than 30% of the time even when internal knowledge would suffice. Recent work on When2Tool goes further: tool necessity is **linearly decodable** from pre-generation hidden states with **AUROC 0.89–0.96**, and a lightweight Probe&Prefill policy cuts tool calls by **48%** with only **1.7%** accuracy loss. In practice, this means your ACI should begin Hell Mode with a classifier that decides among at least four actions: **answer directly, ask a clarification question, retrieve a tool family, or execute a tool**.

The second firebreak should be an explicit **indecisive action space**. Relign argues that many tool hallucinations occur because classic tool-calling setups force the model to choose among only “decisive” actions—i.e., direct tool calls—even when preconditions are not met. Its answer is to add actions like **TalkToUser** and **ChangeTools**, so that the model can defer, clarify, or reroute instead of hallucinating a tool or fabricating parameters. Relign also gives you the right evaluation framing: use **Reliable Pass Rate** to discount successes polluted by hallucinated tool calls, and a **Benefit-cost Utility** that penalizes both hallucinations and excessive tool-call counts. That is exactly the metrics layer Hell Mode needs.

The third firebreak should be a **meta-verification and reflection loop** around every non-trivial call chain. Tool-MVR is the best recent template here. It combines **meta-verification** of APIs, queries, and reasoning traces with reflection learning in an **Error → Reflection → Correction** loop. On StableToolBench it beats ToolLLM by **23.9%** and GPT-4 by **15.3%** while **reducing API calls by 31.4%**; on a reflection benchmark it reaches **58.9%** error correction versus **9.1%** for ToolLLM. So Hell Mode should not simply widen the tool frontier. It should also deepen the **verification stack**: pre-call schema checks, mid-trajectory reflection checkpoints, and post-call plausibility scoring.

The fourth firebreak should be **history compression**. SWE-agent shows that “full history” is worse than a bounded recent window, and WildToolBench reports that instruction transitions can cause **up to 30%** accuracy loss because prior tool-use decisions bias later ones. The mechanism they identify is exactly what you called “tool-use toxicity”: once a model has entered a tool-using groove, long conversational history nudges it to keep calling tools, including redundant ones. In a Max tier, you therefore want **structured state summaries** rather than raw trajectories: active goal, tool family currently in play, outstanding uncertainties, last validated observation, and a small set of verified artifacts. Do **not** stream the whole tool transcript back into the next prompt unless you have a very good reason.

The fifth firebreak should be **typed execution governance**. Relign’s taxonomy is useful operationally: distinguish **tool type hallucination**, **tool timing hallucination**, **tool format hallucination**, and **tool content hallucination**. Then make each class impossible or costly. For example, prevent non-existent tool names by hard whitelisting; prevent timing hallucinations with duplicate-call detection and “same input, same output” suppression; prevent format hallucinations with schema validation before execution; and prevent content hallucinations by checking whether arguments are grounded in user input or tool outputs rather than invented. Those checks can be implemented outside the LLM and should reside in the ACI.

## Architectural verdict

The deepest lesson from this literature is that your Heaven–Hell spectrum should **not** be implemented as a slider over “how many tools are dumped into the prompt.” It should be a slider over **how much freedom the router has**, **how many execution privileges the agent gets**, **how much verification is required**, and **how large the active frontier may become at runtime**. The global tool registry can be enormous; the prompt-visible frontier should remain small. That is the common pattern across ToolLLM, AnyTool, ToolNet, ToolRet, and MCP-style retrieval work.

For a coding-harness ACI, the most defensible default is a **Heaven-biased Native mode**: compact native primitives, concise feedback, bounded history, strong edit/search affordances, and no direct exposure to a wide tool universe. Move upward only when a necessity gate says the task truly needs more reach. Once you move into High or Max, switch from direct serialization to **hierarchical routing** or a **tool graph**, keep the local branch factor near **5–6**, cap active prompt exposure around **32–64**, and require reflection/verification loops for any multi-step or state-changing workflow. That is the architecture most consistent with current evidence.

If you want one sentence to govern the product: **“Maximal capability should live in the retrieval-and-governance substrate, not in the context window.”** The literature is increasingly clear that once tools become numerous, **routing quality, interface compactness, and reliability calibration matter more than raw tool count**.
